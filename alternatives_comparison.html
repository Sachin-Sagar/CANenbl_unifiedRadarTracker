Alternative Implementation Approaches

Comparison of Serialization and Inter-Process Communication (IPC) Methods

Executive Summary

The current implementation relies on a multiprocessing.Manager.dict for IPC and manual casting of non-standard types (like numpy.float64) to standard Python types for serialization. If this approach were to become a bottleneck, several alternatives exist. This document compares these options based on performance, complexity, and ideal use case.

Method Relative Performance Implementation Complexity Best For
Current: Manager.dict + Casting Low-Medium Low Rapid prototyping and moderate throughput scenarios.
Custom JSON Encoder Low Low Handling diverse data types for file logging.
multiprocessing.Queue High Medium A general-purpose, high-throughput IPC channel.
multiprocessing.shared_memory Highest High Extreme real-time needs with zero data copying.
MessagePack / Protobuf Very High Medium-High Minimizing data size and serialization overhead.

Current Implementation: Manager.dict & Manual Casting

The application currently uses a multiprocessing.Manager.dict() for sharing live CAN data between the CAN logger and the radar tracker processes. This object provides a convenient, dictionary-like interface that handles the serialization of standard Python data types automatically.

Current IPC Flow

  • Pros: Simple to implement, dictionary-like API is intuitive and flexible.
  • Cons: The Manager runs in a separate process, which introduces significant serialization (pickling) and communication overhead. It is known to be much slower than other IPC methods. It also fails to serialize non-standard types (like NumPy's), requiring manual data casting before insertion.

Option 1: Smarter Serialization

Custom JSON Encoder

This approach centralizes the logic for handling non-standard data types, making the application code cleaner. Instead of adding float() casts in multiple places, a single encoder class is taught how to handle types like those from NumPy.

  • Pros: Centralizes conversion logic, improves code readability.
  • Cons: Only solves the problem for JSON file output (json.dumps). It does not solve serialization issues for IPC mechanisms like Manager.dict which use pickle.

Option 2: Faster Inter-Process Communication (IPC)

If the Manager.dict becomes a performance bottleneck due to its overhead, switching to a more direct IPC method is the best solution.

A. `multiprocessing.Queue`

A queue provides a simple and robust way to pass messages from a producer (the CAN logger) to a consumer (the radar tracker). It's significantly faster than a managed dictionary.

`multiprocessing.Queue` Flowchart

B. `multiprocessing.shared_memory`

For maximum performance, shared memory allows multiple processes to access the same block of RAM directly, eliminating all serialization and data copying overhead.

Option 3: More Efficient Data Formats

Instead of text-based JSON, using a binary format can significantly reduce data size and the time spent on serialization/deserialization.

JSON

Human-readable text format. Verbose but easy to debug.

{"speed": 95.5}

Size: ~17 Bytes

MessagePack

Binary format, like a compressed JSON. Not human-readable.

0x81 0xA5 speed 0xCB 0x40 0x57 0xE0...

Size: ~15 Bytes

Protocol Buffers

Schema-based binary format. Fastest and smallest, but requires a compilation step.

Schema: `float speed = 1;`
Data: 0x0D 0x00 0x00 0xBE 0x42

Size: ~5 Bytes
