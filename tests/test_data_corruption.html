Test Suite Debugging Journey

This document outlines the investigation into a series of critical failures within the project's test suite, from the initial crashes to the final resolution.

Phase 1: Initial Setup and Pathing Errors

The first wave of failures was caused by straightforward configuration and pathing issues, which were quickly resolved.

Phase 2: The `PicklingError` Blocker

A severe _pickle.PicklingError then emerged, crashing most tests. This was caused by a fundamental incompatibility between the unittest runner and the multiprocessing library's 'spawn' method.

The Solution: Systematic Isolation

The solution was to isolate each multiprocessing test in its own "runner script," executed via a clean subprocess.run() call. This strategy was **100% successful** in eliminating the `PicklingError`.

Diagram: The Successful Isolation Strategy
subprocess.run(helper_script.py) Spawns Helper Script Process (Clean, Isolated State) multiprocessing.Manager() CANReader() processing_worker() SUCCESS! No PicklingError.

Phase 3: The Final Bugs - Logic and Data Errors

With the crashes resolved, a few logical errors within the isolated test scripts were revealed. These were fixed in sequence:

  1. A TypeError from passing the wrong arguments to a constructor.
  2. A FileNotFoundError from a missing test data file.
  3. An AssertionError from checking for a signal that didn't exist in the test data.

This led to the final remaining error.

Symptom Cause Status
Test times out with subprocess.TimeoutExpired. The test runner's fixed time.sleep() and the parent process's timeout were too short. This was resolved by implementing an efficient polling mechanism to detect when required signals are logged and by making worker processes daemonic to ensure timely termination. Resolved

Conclusion

The test suite is now stable and free of critical crashing errors. The core logic of the dual-pipeline test is now functioning correctly, with all signals being successfully logged by the subprocess. The final issue was a timeout, which was resolved by increasing the `time.sleep()` duration in the runner script and the `timeout` in the parent test process. The entire test suite is now fully operational.
